# AGENTS.md

This repository is a public technical portfolio project for decision-support architecture, reporting architecture, data architecture, analytics engineering, and solution-design roles.

## Working rules

- Keep changes narrow and intentional.
- Do not introduce fake client claims, protected data, official data, internal workplace material, or exaggerated outcomes.
- Use only generic, realistic, non-client examples.
- Do not imply official endorsement or access to protected systems.
- Prefer practical architecture documentation over abstract theory.
- Keep the playbook commercially realistic without inventing case-study results.
- Do not turn this repo into a Power BI report, Python engine, or dbt project.
- Keep documentation consistent across the numbered playbook sections.
- Avoid generic tutorial-style content and vague marketing language.

## Architecture standards

- Define the reporting problem before proposing the target state.
- Separate source data, transformation, semantic model, reporting output, review routine, and action follow-up.
- Document ownership for data, KPI definitions, report refresh, review forums, and handover.
- Show where data-quality controls sit in the lifecycle.
- State assumptions and limitations clearly.
- Keep diagrams readable and tied to the written playbook.
- Use Mermaid diagrams where they clarify source-to-output or control-loop thinking.

## Documentation standards

Every major section should explain:

- the architecture problem being addressed;
- the intended reader;
- the relevant decision-support process;
- the source-to-output or control relationship;
- what decisions or actions the artefact supports;
- what is not covered;
- how the artefact would translate to a real organisation.

## Verification

Before considering a task complete, check:

- README accuracy;
- consistency across numbered docs;
- no protected or sensitive data;
- no fake client claims;
- no internal workplace references;
- no unsupported delivery outcomes;
- no broken placeholder sections;
- diagrams match the written architecture;
- limitations are honest and visible.
